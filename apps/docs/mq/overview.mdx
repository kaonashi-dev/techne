The MQ module gives you typed queue contracts, two styles of dispatching, and
a full set of Laravel-grade primitives (chains, batches, uniqueness, per-job
middleware, deferred dispatch) on top of pluggable drivers.

## Quick start — the Dispatchable class

```ts
import { Dispatchable, Queueable, defineQueue, mq } from "@kaonashi-dev/techne/mq";
import { TechneFactory } from "@kaonashi-dev/techne";

export const EmailsQueueDef = defineQueue({
  name: "emails",
  jobs: { sendWelcome: {} as { to: string } },
});

@Queueable()
class SendWelcome extends Dispatchable<{ to: string }> {
  static queue = EmailsQueueDef;
  async handle({ to }: { to: string }) {
    await sendWelcomeEmail(to);
  }
}

const app = await TechneFactory.create({
  plugins: [mq({ queues: [EmailsQueueDef] })],
  providers: [SendWelcome],
});

// Dispatch from anywhere — single statement, fully typed:
await SendWelcome.dispatch({ to: "alice@example.com" })
  .delay(60_000)
  .tries(3);
```

That's the whole minimum loop: declare the queue contract, mark the job class
with `@Queueable()`, dispatch with the fluent builder.

## Two styles, one core

Pick the style that fits your codebase. They interoperate freely.

| Style | When to use | Pages |
|---|---|---|
| **Dispatchable class** (Laravel-style) | One class per job. Co-locates payload type, queue, and handler body. | [Dispatchables](/mq/dispatchables) |
| **Contract API** (`@Processor` + `@On`) | Multiple handlers grouped in one processor class. Mirrors NestJS patterns. | [Contracts](/mq/contracts), [Processors](/mq/processors) |

Both share the same fluent dispatch builder, the same lifecycle hooks, the
same chain/batch/uniqueness/middleware primitives, and the same drivers.

## What's in the box

- **[Contracts](/mq/contracts)** — `defineQueue`, `defineQueueBag`, the
  typed `.dispatchers` map.
- **[Dispatchables](/mq/dispatchables)** — `Dispatchable` base class,
  `@Queueable`, fluent dispatch, class-level defaults
  (`@Tries`, `@Backoff`, `@Timeout`, `@OnQueue`).
- **[Processors](/mq/processors)** — `@Processor` + `@On` for the
  contract-API style.
- **[Lifecycle hooks](/mq/lifecycle)** — `failed()` method + `@OnFailure`
  for cleanup after final retry.
- **[Chains](/mq/chains)** — sequential pipelines via `chain([...]).dispatch()`.
- **[Batches](/mq/batches)** — fan-out + barrier callbacks via
  `batch([...]).then().catch().finally().dispatch()`.
- **[Uniqueness](/mq/uniqueness)** — at-most-one-in-flight via
  `@Unique` / `@UniqueUntilProcessing`.
- **[Job middleware](/mq/middleware)** — `RateLimited`,
  `WithoutOverlapping`, `ThrottlesExceptions`, plus custom middleware.
- **[Deferred dispatch](/mq/after-response)** — `dispatchAfterResponse`
  fires after the HTTP response flushes.
- **[Drivers](/mq/drivers)** — `MemoryDriver` and `RedisDriver`.
- **[Deployments](/mq/deployments)** — running HTTP and workers in
  separate processes.
- **[Legacy `./queue`](/mq/legacy-queue)** — back-compat barrel for
  pre-`./mq` code.

## Low-level usage

You can drive `Queue` and `Worker` directly without the plugin layer when you
need it (tests, scripts, custom integrations):

```ts
import { Queue, Worker } from "@kaonashi-dev/techne/mq";

const queue = new Queue("reports");
await queue.add("build", { reportId: 1 }, { attempts: 3, backoff: 1000 });

const worker = new Worker(queue, async (job) => {
  return { ok: true };
});
await worker.run();
```
