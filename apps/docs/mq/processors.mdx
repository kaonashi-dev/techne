---
title: "Processors"
description: "@Processor + @On for grouping handlers in one class. The contract-API counterpart to Dispatchables."
---

The **processor style** keeps all handlers for a queue in one class.
Decorators bind methods to job names. It's the equivalent of the
[Dispatchable](/mq/dispatchables) style for codebases that prefer fewer
files and more methods per class.

## The pattern

```ts
import { Processor, On, type Job } from "@kaonashi-dev/techne/mq";

@Processor(PayinsQueueDef)
class PayinsProcessor {
  constructor(
    private readonly execution: CoPayinsExecutionService,
    private readonly payinsDao: PayinsDao,
  ) {}

  @On("initiate-payin")
  init(job: Job<{ payinId: string }>) {
    return this.execution.initiatePayin(job.data.payinId);
  }

  @On("payin-expiration-check")
  async checkExpiration(job: Job<{ payinId: string }>) {
    /* … */
  }
}

await TechneFactory.create({
  plugins: [mq({ queues: [PayinsQueueDef] })],
  providers: [PayinsProcessor],
});
```

**`@Processor(def)`** binds the class to a `QueueDef`. **`@On(name)`** binds
a method to a job name on that queue.

## Typed `@On` with `onFor(def)`

For full type safety against the queue's job names (typo → TS error), use
the typed factory:

```ts
import { onFor, type JobOf } from "@kaonashi-dev/techne/mq";

const PayinsOn = onFor(PayinsQueueDef);

@Processor(PayinsQueueDef)
class PayinsProcessor {
  @PayinsOn("initiate-payin")           // ✓ TS-checked against the def
  init(job: JobOf<typeof PayinsQueueDef, "initiate-payin">) { /* … */ }

  @PayinsOn("not-a-real-job")           // ✗ TS error
  bad() {}
}
```

## Dispatching to a `@Processor`

The contract-API dispatchers map is what producers use:

```ts
const { initiatePayin } = PayinsQueueDef.dispatchers;

await initiatePayin({ payinId }).delay(60_000).tries(3);
```

See [Contracts](/mq/contracts#the-dispatchers-map) for the full surface.

## Auto `handle()` fallback

For single-job-type queues, you can skip `@On` and write a `handle()`
method — it's auto-registered as the default for any job name on the queue.

```ts
import { Queue } from "@kaonashi-dev/techne/mq";

@Queue("emails")
class EmailProcessor {
  async handle(job: Job<{ to: string }>) {
    await sendEmail(job.data);
  }
}
```

`@Queue(name)` is an alias for `@Processor`/`@MqProcessor`. If both `@On(name)`
handlers and `handle()` exist, named handlers win.

## When to use Processors vs Dispatchables

| | Processor | Dispatchable |
|---|---|---|
| Handlers per class | Many | One |
| File count | Fewer | More |
| Pairs naturally with | Contract API + dispatchers map | Class-level decorators (`@Tries`, `@Backoff`, etc.) |
| Failure lifecycle | `@OnFailure(name)` | `failed()` method |
| Per-job middleware | (not directly — wrap the body manually) | `middleware()` method |

Both styles coexist on the same queue, **but each job name must have
exactly one handler**. Duplicates throw at startup.

## Legacy aliases

`@MqProcessor(name)`, `@MqProcess(jobName)`, and `@InjectMq(name)` are the
older, string-keyed forms. Functionally equivalent — the new decorators are
thin wrappers. Prefer `@Processor(def)`, `@On(name)`, and `@InjectQueue(def)`
because they're tied to the typed `QueueDef`.
